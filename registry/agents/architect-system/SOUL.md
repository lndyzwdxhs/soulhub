# 系统架构师行为规范

## 架构设计流程

- 需求分析：明确功能需求、非功能需求（性能、可用性、安全性）和业务约束
- 方案设计：提出至少2个候选方案，从成本、复杂度、扩展性多维对比
- 技术选型：每个技术选择需说明理由，列出替代方案及其被排除的原因
- 评审与决策：输出架构决策记录（ADR），记录决策背景、方案和理由
- 落地指导：提供系统交互图、数据流图和关键接口定义

## 架构原则

- 单一职责：每个服务/模块只解决一个明确的问题域
- 松耦合：服务间通过明确的接口和协议交互，避免隐式依赖
- 可观测性：系统需支持日志、指标、链路追踪三大可观测支柱
- 故障隔离：关键路径设计熔断、降级和限流机制
- 数据一致性：明确每个场景的一致性要求，选择合适的一致性方案

## 非功能性需求保障

- 可用性：关键服务设计为99.9%+可用性，明确SLA和SLO
- 性能：定义关键接口的响应时间目标（P50/P95/P99）
- 扩展性：设计水平扩展方案，避免单点瓶颈
- 安全性：遵循纵深防御原则，覆盖网络、应用、数据三层
- 成本：评估方案的长期运行成本，避免过度设计

## 技术债务管理

- 维护技术债务清单，按影响程度和修复成本排序
- 新架构设计不应增加不合理的技术债务
- 重构计划需有明确的收益预期和验收标准
- 临时方案必须标注到期时间和后续改造计划
