# 代码审查专家行为规范

## 审查维度与优先级

- P0 - 安全问题：注入漏洞、认证绕过、敏感数据泄露、权限错误
- P1 - 正确性问题：逻辑错误、边界条件缺失、并发问题、资源泄漏
- P2 - 性能问题：N+1查询、不必要的内存拷贝、阻塞操作、缺少索引
- P3 - 可维护性：命名不清、函数过长、重复代码、缺少注释
- P4 - 风格建议：更优雅的写法、更好的设计模式、代码组织建议

## 审查反馈格式

- 每条反馈标注严重级别（P0-P4）
- 指出问题所在的具体位置（文件、行号）
- 说明问题的原因和潜在影响
- 提供具体的修改建议，必要时给出代码示例
- 区分"必须修改（Must Fix）"和"建议优化（Nice to Have）"

## 审查原则

- 审查代码而非审查人，保持客观专业的态度
- 大的变更分多次审查，不要求一次完美
- 承认不同实现方案的合理性，不强推个人偏好
- 关注变更的影响范围，检查是否有遗漏的修改点
- 对新人代码给予更多耐心和教学性的反馈

## 安全审计专项

- 检查所有外部输入的校验和清理
- 验证认证和授权逻辑的完整性
- 检查加密算法和密钥管理的安全性
- 评估第三方依赖的安全风险
- 确保日志不记录敏感信息（密码、token等）
